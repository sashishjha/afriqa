# AfriQA Competition Compliance README

## Competition

Multilingual Health Question Answering in Low-Resource African Languages Challenge

Hosted on Zindi in partnership with ITU.

Goal:
Build a multilingual AI system that answers maternal, sexual, and reproductive health (MSRH) questions in low-resource African languages. The system must generate answers in the same language as the input question.

---

# Dataset Summary

Training Data:

* ~29,815 training examples
* ~6,686 validation examples
* Question-answer pairs

Languages:

* Akan
* Amharic
* Luganda
* Swahili
* English

Nine language-country configurations are represented.

Test Data:

* 2,618 questions
* Only questions are provided
* Answers must be generated by the model
* Evaluation is performed against hidden reference answers.

---

# Primary Objective

For every test question:

Input:

* Health question

Output:

* Fluent answer
* Correct answer
* Same language as input
* Focused on maternal, sexual, and reproductive health topics.

---

# Evaluation

Leaderboard score is a weighted combination of:

1. ROUGE-L F1
2. ROUGE-1 F1
3. LLM Judge score

Current leaderboard displays:

* Multi Score
* ROUGE-L F1
* ROUGE-1 F1
* LLM Judge score

Public leaderboard is calculated on only part of the hidden test set until competition closes. Final ranking may differ. (Shown on competition leaderboard page.)

---

# Submission Requirements

Submission file must:

* Contain predictions for every test example.
* Preserve row order exactly.
* Contain one generated answer per question.
* Follow competition sample submission format.
* Not contain missing values.
* Not contain extra rows.
* Not contain reordered rows.

Before every submission:

Checklist:
[ ] Correct number of rows
[ ] No empty predictions
[ ] UTF-8 encoding
[ ] No accidental truncation
[ ] Same ordering as test file
[ ] Submission successfully loads in validation script

---

# Hard Constraints

The agent MUST:

1. Answer in the same language as the question.
2. Avoid language switching.
3. Avoid English leakage into African-language outputs unless medically necessary.
4. Generate complete sentences.
5. Remain medically relevant.
6. Stay concise but informative.
7. Preserve meaning during translation-style generation.
8. Avoid hallucinating unrelated information.
9. Avoid generic filler text.

---

# Competition-Safe Development Rules

## Allowed

* Use pretrained open-source models.
* Fine-tune models.
* Perform multilingual training.
* Use retrieval-augmented generation if competition rules permit.
* Use public resources.
* Use public multilingual checkpoints.
* Use public translation resources.
* Use parameter-efficient fine-tuning (LoRA, QLoRA, etc.).

---

## Must Avoid

Unless explicitly allowed in official rules:

* Manual labeling of test answers.
* Looking for hidden test references.
* Data leakage.
* Using private datasets that violate licensing.
* Reverse-engineering leaderboard labels.
* Human-written answers for test samples.
* Any attempt to identify hidden ground truth.

# RBCCPS Cluster Rules (User Environment)

Follow:

1. Interactive validation first

   * salloc
   * srun

2. Verify:

   * Dataset loading
   * Model loading
   * Training starts successfully

3. Submit production run using:

   * sbatch

4. Monitor:

   * squeue
   * scontrol show job
   * log files

5. Use checkpoints because cluster runtime limit is 24 hours.

6. Respect user quota:

   * Max 3 GPUs
   * Max 1 running job
   * Max 2 total jobs

# Agent Success Criteria

A model is considered successful if it:

* Produces medically relevant answers.
* Maintains correct language.

* Remains reproducible.
